# Postmortem: 随机超时与 429 导致请求抖动

## 事件摘要

- 场景：外部 API 在高峰期出现随机 `read timeout` 与 `429`。
- 影响：请求成功率下降，重试次数增多，端到端耗时波动。

## 根因

1. 上游服务短时限流与网络抖动叠加。
2. 客户端缺少并发保护，瞬时请求峰值进一步放大 429。
3. 连续失败时仍持续打上游，造成雪崩风险。

## 修复措施

1. 重试策略可配置：按错误码与状态码决定是否重试。
2. 加入 `Semaphore` 并发上限，平滑请求峰值。
3. 引入 `CircuitBreaker`：连续失败超过阈值后快速失败，保护上游。
4. 增加故障注入测试：随机 timeout/429，验证恢复能力。

## 验证

- 单测覆盖：
  - 熔断打开后快速失败
  - 并发上限不超过配置值
  - 故障注入场景可恢复到成功
- 本地执行：`uv run python -m pytest -q`

## 后续改进

- 将熔断状态与失败率接入指标系统。
- 为不同上游服务配置独立重试与熔断参数。
